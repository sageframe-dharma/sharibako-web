Frequently asked questions

Straight answers — including the ones that admit limits.

In the order people actually ask them. Where a question carries a common misconception, the answer starts by correcting it.

Is my .env encrypted after I import it into Sharibako?

No — and that's deliberate. Your tools need to read that file; encrypting it would break them or require decrypting it constantly, which protects nothing on a live machine. Importing makes an encrypted copy in the vault. The .env in your project stays exactly as it was — except now it's disposable, because the vault can regenerate it.

Then what's the point?

Before Sharibako, each .env was the only copy of its secrets — unbackable, unversioned, unrotatable, one git add . from disaster. After, the canonical copy is encrypted, versioned, backed up to a remote that only sees ciphertext, and shared keys live once instead of eight times. The working file goes from single point of failure to cache.

What if someone steals my laptop?

FileVault — Mac full-disk encryption, on by default — is your protection there, for everything on the disk, .env files included. Sharibako adds that the vault's contents are ciphertext even inside the disk image, and your decryption key is in the Keychain, which the thief can't open without your fingerprint or password.

What if someone is sitting at my unlocked machine?

Then they are you, as far as the machine is concerned — they can read your browser sessions, your SSH keys, and any .env file. No dev-secret tool can protect a live, unlocked session, and any that claims to is selling theater. What Sharibako gates behind Touch ID is the vault: revealing a stored secret's value, decrypting for materialize or run. The blast radius of an unlocked machine is your working files — not your whole vault.

Is it really safe to push my secrets to GitHub?

The remote only ever receives age-encrypted files — ciphertext. Without your private key (which never leaves your machine and Keychain), the repo's contents are noise. Use a private repo anyway: defense in depth, and the names of your scopes and keys are metadata you may not want public. One honest caveat: git history retains old ciphertext forever, so treat "delete from vault" as "stop using," not "erase from time."

What happens if I lose my key?

You lose the vault. That is the deal with real encryption — there is no recovery email. This is why first-run makes you save a backup copy of the key and verifies you actually did. Keep that backup somewhere durable: iCloud Drive, a printed copy, a password manager. Vault + key = everything; vault − key = nothing.

What happens if I lose the vault?

If you configured a remote: git clone, restore your key, materialize — you're back. If you never set a remote and the disk is gone, the vault is gone. Set the remote. It's one field in first-run.

Where does my key live day-to-day?

In the macOS Keychain, stored so that reading it requires user presence — Touch ID or your password. When Sharibako needs it, it asks, uses it, and scrubs the temporary copy. A successful Touch ID is reused for five minutes so bulk operations don't prompt you per secret.

What counts as a secret? Should I import everything from my .env?

Import anything you'd have to rotate if it leaked. Skip pure configuration — ports, hostnames, flags — that belongs in git, visibly. Sharibako's import lets you decide per key: import it, mark it shared, leave it alone, or skip it.

Two of us share a project. Can we share the vault?

Two humans who trust each other can share the vault repo and the key — it works fine. What Sharibako doesn't have is team features: no per-person access, no audit log of who read what, no revoking one person. When you need those, you've outgrown the tool — use a team secret manager.

What about my servers / CI / production?

Out of scope, honestly. run and materialize are for humans at dev machines. Production secrets belong in your platform's secret manager — AWS Secrets Manager, Vault, your PaaS's env config — which exist precisely for unattended machines. Sharibako is for the laptop end of the pipeline.

How is this different from 1Password?

A password manager stores secrets; it doesn't know what a project is. Sharibako is project-shaped: it knows which directory maps to which secrets, writes and regenerates .env files, detects drift, injects into processes, and de-duplicates keys across projects. Keep your logins in the password manager; keep your project secrets somewhere that understands projects. (And keep your key backup in the password manager — good pairing.)

What's a "scope"?

One project's slice of the vault — usually one repo ↔ one scope. The .sharibako marker file in your project directory records which scope it binds to and where the .env should be written. The marker contains no secrets.

What does "drift" mean?

Your .env says one thing, the vault says another — usually because you hand-edited the file. Check Drift shows exactly which keys differ, and you choose the winner: reconcile from the vault, or re-import the file's value.

Is Sharibako open source? Why would I pay?

Fully open source, GPL-3.0 — a security tool you can't read is a tool you're taking on faith. The paid download buys you the signed, notarized, ready-to-run build (no toolchain, no Gatekeeper fights) and supports development. Same code either way.
