For the developer with fifteen projects and one drawer of API keys

One encrypted box for every API key you own.

Sharibako is a native Mac app and command-line tool that keeps your projects' secrets in a single encrypted vault behind Touch ID. Your .env files keep working exactly as they do today—they just stop being the only copy of your secrets.

The problem

Every project has a .env. Each one is the only copy.

Your secrets—API keys, tokens, database passwords—live in little gitignored files scattered across your projects. Gitignored means unversioned. Unversioned means unbackable. The file that matters most is the one your backups are explicitly told to skip.

LOSS

Laptop dies, keys die with it

Rebuilding your .env files is a scavenger hunt through provider dashboards, old notes, and Slack messages. Some keys you'll never find again.

LEAK

Backing them up is the leak

Dropbox, Time Machine, a private "dotfiles" repo—every obvious fix copies plaintext secrets into systems you can't audit. And one git add . puts them in history forever.

SPRAWL

One key, pasted eight times

The same OPENAI_API_KEY lives in eight projects. Time to rotate it? Good luck finding all eight. Nothing tracks where your keys went.

The fix

Every secret lives once. Everything else regenerates.

Sharibako keeps one canonical, encrypted copy of each secret in a vault — a plain folder in your home directory where every key is a small age-encrypted file, and the whole folder is a git repository.

What you get

Three things a folder of .env files can never give you.

Never lose a key again

The vault is a git repo. Push it to a private remote and your secrets are backed up, versioned, and recoverable—as ciphertext, so the remote learns nothing. A new laptop is: clone the vault, restore your key, materialize.

Rotate once, not eight times

A key used by many projects lives in the vault exactly once; projects link to it. Change it in one place and every project picks up the new value on the next materialize. Sharibako also detects drift — when a .env on disk no longer matches the vault—and shows you exactly which keys differ.

Or skip the file entirely

sharibako run decrypts your secrets in memory, hands them to your process as ordinary environment variables, and leaves nothing on disk. Your code can't tell the difference.

What Sharibako doesn't do

We don't encrypt your working .env—on purpose.

Your tools need that file in plaintext, and anyone who controls your live, unlocked session can read whatever your tools can read. Encrypting the working copy inconveniences you daily and an attacker not at all — that's security theater, and plenty of tools sell it.

Sharibako protects every other copy instead: the backup, the sync, the git remote, the history, the next machine. Those are the copies that actually leak. The full argument is here—it's the most important page on this site.

Two surfaces, one vault

A calm Mac app. A complete CLI. Your choice, or both.

The Workshop—the native Mac app—gives you your whole vault at a glance: every project, every key, what's in sync and what's drifted, with a first-run setup that takes minutes and explains itself. Reveal a value with Touch ID, rotate it, materialize, sync—all without a terminal.

The CLI does everything the app does, scriptably:

$ sharibako init # bring this project's .env into the vault $ sharibako materialize # write .env from the vault $ sharibako run npm start # inject secrets, no file at all $ sharibako rotate newsletter MAILCHIMP_API_KEY $ sharibako status # every project, every key, at a glance $ sharibako sync # commit + push ciphertext to your remote

Both surfaces read the same vault and the same Touch ID-guarded key. Set up in the app, script in the terminal—or never open the terminal at all.
