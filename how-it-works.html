The deep dive

How Sharibako works — from zero.

Everything on this page assumes nothing. If you've shipped secrets for years, skim the headers and stop at the theater argument — it's the part that explains every design decision. If .env is new to you, start here.

What is a secret?

A secret is a value that grants a capability. An API key that lets code send email as you. A database password. A token that posts to your Slack. If someone else had the value, they could do the thing — that's what makes it a secret.

The litmus test: if this value leaked, would you have to change it? If yes, it's a secret. If no — a port number, a feature flag, a public URL — it's just configuration. Configuration wants to be visible and in git. Secrets are the part that can't be.

ValueSecret?Why
API key, access tokenYesGrants capability; rotate on leak

Publishable / public API keyNoDesigned to be visible client-side

Gray zones exist — OAuth client IDs, internal hostnames, anything with "test" in the name (test keys still bill and leak). When unsure, apply the rotation test.

How .env files work

Nearly every modern project keeps its secrets in a file called .env at the project root — plain text, one KEY=value per line:

MAILCHIMP_API_KEY=a1b2c3...
DATABASE_URL=postgres://admin:hunter2@db.example.com/prod

When the app starts, a small library (dotenv and its hundred cousins) reads the file and turns each line into an environment variable — a named value the running program reads with os.environ["MAILCHIMP_API_KEY"] or process.env.MAILCHIMP_API_KEY. The app never knows the file existed; it just sees the variables. By convention .env is listed in .gitignore, so secrets stay out of version control while the code goes in.

This is a genuinely good convention, and Sharibako doesn't replace it. It keeps it — and fixes what's wrong with it.

The family, precisely: .env holds the real values, gitignored and machine-local. .env.local is the same idea; some frameworks load it over .env for per-machine overrides. .env.example is the keys with values stripped, committed as documentation of what a project needs. And .env.production-style variants are, historically, the single most common way real secrets end up in git — because ignore patterns miss them.

Where .env files fail

The naive model says: my secrets are in a gitignored file, so they're handled. Actually, four problems are quietly accumulating:

  1. Each .env is the only copy. Gitignored means unversioned, and unversioned means unbackable-up — deliberately. If your laptop dies, every .env on it dies too. The file that matters most is the one your backup discipline is explicitly told to skip.
  2. Backing them up creates the leak. The obvious fix — Dropbox, Time Machine, a private dotfiles repo — copies plaintext secrets into systems you don't control and can't audit. The secret now exists in sync history, on a drive in a drawer, in a repo you'll one day make public without remembering what's in it.
  3. Git is one mistake away. One git add . before the ignore entry exists and the secret is in history — where deleting the file does not remove it. Every clone made in the meantime keeps the old value forever. Leaked-key scanners on GitHub stay busy for a reason.
  4. Sprawl and drift. The same key pasted into eight projects; rotation becomes a manhunt. No answer to "what was the old value," "when did this change," or "which projects use this key."

Why encrypting your .env is theater

Here is the correction that shapes Sharibako's entire design.

The instinctive fix for "plaintext secrets on disk" is to encrypt the .env itself. Actually, that protects almost nothing — because your tools need the plaintext to run. Whatever encrypts the file must decrypt it every time npm start runs, which means the plaintext (or the means to produce it) is always present and reachable on a live machine.

And anyone who can act as you on your live machine — malware running in your session, a person at your unlocked keyboard — can read everything you can read: your SSH keys, your browser cookies, your password manager while it's unlocked, and yes, your secrets, encrypted file or not. Encryption at that layer inconveniences you daily and an attacker not at all. That is the definition of security theater.

What encryption genuinely protects is every copy that leaves your live session: the synced copy, the backed-up copy, the pushed copy, the historical copy, the copy on the drive in a stolen laptop (alongside full-disk encryption, which is the real answer to theft). Those copies sit in places where an attacker doesn't need your live session — and those are exactly the copies that leak in practice.

Let the working copy be plain. Make every other copy safe.
That single sentence is Sharibako.

The architecture

Sharibako keeps one canonical, encrypted copy of every secret in the vault — by default a folder at ~/.sharibako/vault. Three properties define it:

  • The filesystem is the database. Every secret is a small file encrypted with age — a modern, audited, deliberately simple encryption tool. A scope (one project's slice) is a directory. You can look inside the vault with ls and understand everything you see. No opaque database, no proprietary format, no lock-in: the vault is readable by standard tools forever.
  • Git is the backup. The vault folder is a git repository. sync commits and pushes to your private remote — and since every file in it is ciphertext, the remote learns nothing. History comes for free: every rotation is a commit.
  • The Keychain guards the key. The one secret that can't encrypt itself — your age private key — lives in the macOS Keychain, stored so that reading it requires user presence: Touch ID or your password. Decrypt operations prompt; a successful check is reused for five minutes so bulk operations don't nag.

Scopes and markers

A scope is one project's slice of the vault — usually one repo ↔ one scope. A small .sharibako marker file in your project directory records which scope the directory binds to and where its .env should be written. The marker contains no secrets — it's an address, safe to look at, cheap to recreate.

Materialize and run

Materialize decrypts a scope's secrets and writes the project's .env — plain, because your tools need it, and disposable, because the vault can always write it again. Run (sharibako run npm start) skips the file: secrets are decrypted in memory, set as environment variables, and the Sharibako process becomes your command — signals, exit codes, and Ctrl-C all behave normally, and nothing ever touches disk.

Shared secrets and links

A key used by many projects lives once in the vault's shared pool; each project holds a link — a pointer, not a copy. Rotate the shared value once and every linked project materializes the new value. This is the answer to sprawl.

Drift

If you hand-edit a .env, it now disagrees with the vault. Sharibako calls that drift, detects it per-key, and lets you choose the winner — reconcile from the vault, or import the edited value back in. Nothing is ever silently overwritten.

The threat model, stated fully

A security tool owes you a plain statement of what it does and doesn't protect against. Here is Sharibako's.

Protected against:

  • Loss of a working .env — it's regenerable.
  • Loss of the machine — clone the vault from your remote, restore your key backup, materialize.
  • Leakage through backup and sync channels — only ciphertext ever leaves the machine via Sharibako.
  • Git accidents involving the vault — its contents are ciphertext.
  • Stolen powered-off hardware — FileVault plus vault ciphertext plus a Keychain-guarded key.
  • Casual value exposure — secrets are masked in the app until a Touch ID reveal.

Explicitly not protected against:

  • An attacker with your live, unlocked session — they are you, and nothing at this layer can help. (No dev-secret tool can; the honest ones say so.)
  • A compromised tool that legitimately receives the secret — it needs the secret to work.
  • Plaintext copies you make by hand outside Sharibako's flows.
  • "Delete from history" — git retains old ciphertext, so revoke-and-rotate is the real remediation for a compromised key. Everywhere. Always.

The alternatives, honestly

Every approach to secrets is a different answer to "where does the canonical copy live, and what protects it?"

ApproachCanonical copyHonest assessment
Plaintext .env onlyThe file itself Zero setup, universally understood — and unbackable, unversioned, sprawling, one mistake from git history. Fine for a throwaway; a liability by the second project.
Password manager (1Password, Bitwarden)Their encrypted store Excellent custody, project-blind: no scopes, no .env generation, no drift detection. Ideal for logins — and for backing up your Sharibako key.
Cloud secret managers (AWS, Vault, PaaS config)A server Built for unattended machines and teams: IAM, audit logs, rotation policy. Exactly right for production; friction and overkill for a personal laptop. Sharibako deliberately doesn't compete here.
Encrypted-in-repo (SOPS, git-crypt, raw age)Ciphertext in git The right architecture — the one Sharibako shares — delivered as composable primitives. No project binding, no regenerate loop, no drift detection, key custody on you. Great if you enjoy assembling it; Sharibako is that architecture with the plumbing done and Touch ID on the door.

Where Sharibako sits

Deliberately narrow: local-first custody of development secrets, for one human (or two who share everything), on machines they sit at. The canonical copy is files on your disk. Protection is age encryption with the key in your Keychain. Backup and history are git, because you already trust git. It ends where unattended machines and teams begin — by design, those problems belong to the server-shaped tools above.
