Help

Every workflow, step by step.

The app explains itself as you go — this page is the same knowledge in one place, for reading ahead or finding your way back.

Install

Direct download: buy the signed, notarized app, open the .dmg, drag Sharibako to Applications, launch. No Gatekeeper warnings, no setup beyond first-run.

From source (free): clone the repo and follow the README's build instructions. You'll need the Swift toolchain and age (brew install age). Same app, same CLI — you compile it.

Sharibako uses age for encryption and git for the vault's history. First-run checks for both and gives you the copy-paste install line if either is missing.

First-run setup

The first time you launch Sharibako with no vault, a setup guide walks you through five short steps. It takes a few minutes, and the order matters — each step builds on the last.

  1. Prerequisites. Sharibako checks that age (the encryption tool) is installed. If not, it shows the exact brew install age line and a Re-check button. Nothing proceeds without it — even importing an existing key needs age for every later encryption.
  2. Your key. Generate a fresh encryption key (stored in the macOS Keychain behind Touch ID) — or import an identity file if you already have one from another machine. If a Sharibako key already exists in your Keychain, this step says so and moves on. There is never a second key.
  3. Back the key up. The one non-negotiable step. You save a backup copy of the key to a location you choose — iCloud Drive and Documents are suggested — and Sharibako verifies the saved file matches before letting you continue. Lose the key, lose the vault; this backup is your recovery path.
  4. Where you keep code. Pick the folder your projects live under (Sharibako suggests likely candidates like ~/Projects). This becomes the scan root — where Sharibako looks for projects and their .env files. The vault itself lives at ~/.sharibako/vault; this page tells you that plainly.
  5. Optional remote. Paste a private git remote URL (a fresh empty GitHub repo works perfectly) and the vault will back itself up there — ciphertext only — every time you Sync. Skippable, but this is the step that makes your secrets survivable, so: don't skip it.

Finish creates the vault and, if it found projects with .env files under your root, offers to import the first one immediately.

The tour

Help → Welcome to Sharibako… replays the setup guide as a read-only tour against your real configuration: your actual vault path, scan roots, and remote, with live prerequisite checks. It changes nothing — it's the "wait, how does this work again?" button.

Import your projects

Click Ingest Project… in the action panel. Sharibako walks your scan roots and lists every project that has a .env (or .env.local) but isn't in the vault yet — name and path, click to import.

  1. Pick a project from the list (or Choose Other… for one outside your roots).
  2. Review its keys. Each detected secret shows its name and value — in the clear, because they come from a file that's already readable on your disk — and gets a verdict (below).
  3. Confirm the scope name (suggested from the folder name) and type.
  4. Click Import. One Touch ID encrypts everything into the vault and drops a .sharibako marker in the project.

After each import, the list returns with the remaining projects — so you can walk your whole root in one sitting. Your .env files are never modified by import: byte-for-byte untouched.

Re-importing a project that's already in the vault is safe: Sharibako shows only the keys the vault doesn't have yet ("reconcile"), and tells you plainly when there's nothing new.

The five verdicts

Every key you import gets one decision:

VerdictMeaningUse when
Import as scope-localEncrypted copy in this project's scopeThe default — a key this project owns
Move to sharedKey moves to the shared pool; this project links to itA key other projects will use too
Link to sharedThis project points at an existing shared keyThe key is already in the shared pool
Leave aloneStays in the .env, vault ignores itValues that aren't really secrets (ports, flags)
SkipNot imported, not trackedJunk, dead keys, things you'll deal with later

Adding by hand

Not everything arrives via import. The action panel's Add group covers the rest:

  • Add Scope — a new empty project slice, for a project that doesn't have a .env yet.
  • Add Secret — one new key in the selected scope. Opens in its own small window, so you can check other secrets while you type.
  • Add Shared Secret — a new key in the shared pool, ready for projects to link to.

Orphaned markers

An orphaned marker is a .sharibako file in some project directory that names a scope your vault doesn't have — usually a leftover from a deleted scope, or a marker that arrived with a cloned repo. They appear in the sidebar under "Unlinked markers," each with two buttons:

  • Create Scope from Marker — imports that project under the marker's recorded scope name. The healing move.
  • Remove Stray Marker — deletes the marker file (just the marker — the project's .env and the vault are untouched). Asks for confirmation and names the exact file first.

Materialize & preview

Select a scope and click Materialize: Sharibako decrypts that scope's secrets (one Touch ID) and writes the project's .env where the marker says it belongs. If the file already exists and differs, you get a per-key diff and nothing is overwritten until you confirm.

Preview .env shows exactly what would be written — before anything touches disk. Materialize All Stale reconciles every drifted project in one confirmed batch, one Touch ID for the lot. Jump to Directory opens the project folder in Finder.

Run — secrets with no file at all

From a project directory in the terminal:

cd ~/Projects/newsletter
sharibako run npm start

Sharibako reads the project's marker, decrypts its secrets in memory (Touch ID), sets them as environment variables, and then becomes your command — signals, Ctrl-C, and exit codes all behave exactly as if you'd run it directly. Nothing is written to disk. Your code just reads process.env / os.environ as always.

Use run when you'd rather not have a plaintext .env sitting in the project at all; use materialize when a tool insists the file exist.

Reveal, edit, rotate

Secrets are masked in the app until you reveal them — Reveal costs a Touch ID and shows the value until you select something else. Editing a value or its notes prompts the same way. Rotate replaces a value with a new one; the old value is retained in the vault's git history (encrypted), so "what was it before Tuesday?" always has an answer.

A successful Touch ID is reused for five minutes, so a burst of operations — a batch materialize, several reveals — costs one prompt, not ten.

Drift & reconcile

Drift is when a project's .env on disk disagrees with the vault — you hand-edited the file, or rotated a key and haven't re-materialized. Check Drift sweeps every project (one Touch ID) and marks each drifted key in red, per key, per project.

To resolve: Materialize (vault wins — rewrites the file), or re-import the project (file wins — the edited value comes into the vault). Nothing is ever silently overwritten in either direction.

Sync & rescan

Sync commits the vault's current state and pushes to your remote — this is the moment your secrets become survivable. Sharibako never commits behind your back: rotations, imports, and deletions sit uncommitted until you Sync, so the commit boundary is yours.

Rescan re-walks your scan roots — after adding projects, moving directories, or editing the config — and refreshes every project's status glyph in the sidebar.

Shared secrets & links

The same OPENAI_API_KEY in eight projects should be one secret, not eight copies. In Sharibako it lives once in the shared pool; each project holds a link — a pointer, not a copy. Materialize resolves the link and writes the current shared value into that project's .env.

  • During import, choose Move to shared for the first project and Link to shared for the rest — Sharibako suggests the link when it recognizes a matching name.
  • Rotate the shared value once; every linked project gets it on next materialize (drift shows you which ones are stale).
  • Unlink turns a link back into a local copy — the project keeps the current value, independence restored.

Backup & recovery

Two things constitute your entire secret store: the vault (backed up by Sync to your private remote) and the key (backed up during first-run to a file you chose). Everything else — every .env, every marker — regenerates from those two.

New Mac, step by step:

  1. Install Sharibako and age.
  2. Clone your vault: git clone <your-remote> ~/.sharibako/vault
  3. Launch Sharibako → first-run detects no key → choose Import and point it at your key backup file. The key goes into the new Mac's Keychain.
  4. Materialize each project as you need it. Done.

Lost the key backup but the Mac still works? Make a new backup now — the key is still in your Keychain. Lost the key entirely? The vault is unrecoverable — this is real encryption, there is no override. Rotate the underlying credentials at each provider and start a fresh vault. (This is why first-run refuses to continue until the backup is verified.)

Deleting things

Every deletion asks for confirmation, names exactly what will be removed, and touches only the vault:

  • Delete Secret — removes one key from a scope.
  • Delete Scope — removes a project's whole slice from the vault. The project's .env and marker are left in place (the marker will surface as orphaned — remove it there if the project is truly gone).
  • Deleting a shared secret (CLI) — refused if projects still link to it, and the refusal names them. Unlink first, or force it and let the orphaned links surface for healing.
  • Clean (CLI) — removes a project's materialized .env, for when a project goes dormant and you don't want plaintext sitting around.

Deletions, like everything else, are uncommitted until Sync — and git history retains old encrypted values, so deletion means "stop using," not "erase from time." For a compromised key, the remediation is always revoke-and-rotate at the provider.

Troubleshooting

"age not found" at first-run

Install it with brew install age (the wizard shows this line and a Re-check button). If you don't use Homebrew, any install that puts age and age-keygen on your PATH works.

Touch ID prompts more than you expect

Every decrypt needs the key, and the key needs your presence. A successful check is reused for five minutes; batch operations (Check Drift, Materialize All Stale, chained imports) are designed to ride one prompt. If prompts feel excessive during a workflow, that's feedback we want.

A project shows as drifted and I don't know why

Check Drift lists the exact keys that differ. The usual causes: you hand-edited the .env, or rotated a key and haven't re-materialized. Preview .env shows precisely what materialize would write before you commit to it.

An "Unlinked marker" appeared

Some directory has a .sharibako marker naming a scope the vault doesn't have — a leftover, or a marker that came along with a cloned repo. Use Create Scope from Marker to adopt it, or Remove Stray Marker to clear it. Neither touches the project's .env.

The vault won't sync

Sync needs a configured remote (check the sidebar footer) and a reachable network. The tour (Help → Welcome to Sharibako…) shows your current remote. Adding one later: set origin on ~/.sharibako/vault like any git repo.

I built from source and Touch ID doesn't work

Keychain access requires the app to be signed and properly installed — a bare debug build can't reach it. Use the repo's install script, or run with a file-based key (--age-key) during development. The README covers both.
